# Skill — 技能集合

本文件夹包含可直接使用的 SKILL 文件，用于指导 AI 代理在特定场景下以经过验证的最佳实践工作。

---

## 📖 什么是 SKILL？

**SKILL** 是一种可复用的提示词剧本（Playbook），以 `SKILL.md` 文件形式存在。每个 SKILL 文件包含：

| 组成部分 | 说明 |
|----------|------|
| **YAML 头信息** | `name`（技能名称）和 `description`（触发条件描述） |
| **行为规则** | 代理必须遵循的约束和铁律 |
| **执行流程** | 按顺序完成的检查清单和步骤 |
| **输出规范** | 最终交付物的格式和内容模板 |

---

## 📦 本仓库包含的 SKILL

### analyze-flutter-project

对 Flutter 项目进行系统性代码分析，自动生成结构化的 `Project_Info.md` 文档。

- **文件位置：** [`analyze-flutter-project/SKILL.md`](./analyze-flutter-project/SKILL.md)
- **原理：** 只读分析 Flutter 项目的目录结构、配置文件（`pubspec.yaml`、`build.gradle`、`Podfile`）和源码，提取技术栈信息，整理为中文 Markdown 文档
- **输出：** 项目根目录下的 `Project_Info.md`

#### 🎯 触发条件

当用户说出以下任何一句话时自动激活：

| 触发词 | 示例 |
|--------|------|
| 分析类 | 「帮我分析这个 Flutter 项目」、「分析这个项目」 |
| 架构类 | 「看看这个项目的架构」、「看看这个项目结构」 |
| 文档类 | 「生成项目信息文档」、「写 Project_Info.md」 |
| 技术类 | 「这个项目用了什么技术」 |
| 通用类 | 任何涉及 Flutter 项目代码分析和文档生成的请求 |

#### 🛠️ 使用场景

| 场景 | 说明 |
|------|------|
| **接手旧项目** | 第一次接触一个 Flutter 项目时，快速了解其架构、依赖和技术栈 |
| **代码审查** | 审查前自动生成项目概览，了解状态管理方案、路由设计、网络层结构 |
| **文档维护** | 为项目生成标准化的技术文档，便于团队 onboarding |
| **技术评估** | 评估一个 Flutter 项目使用了哪些第三方库和最佳实践 |

#### 📋 分析内容

项目结构 → 状态管理方案 → 网络请求层 → 路由设计 → 主题系统 → 多语言支持 → 工具类/扩展 → 第三方依赖 → 代码规范 → Assets 资源配置

#### ⚠️ 约束

- **只读不写** — 不修改任何项目代码
- **每个结论必须有代码依据** — 标注 `(文件路径:行号)`
- **不确定就问** — 不猜测、不编造
- **大型项目** — 最多读 4 个核心文件加环境配置，不超量读取

## 🔧 安装与卸载

### 安装

将 SKILL 文件放入 AI 代理的 skills 加载目录即可。例如复制到项目或全局 skills 目录：

```bash
cp ./analyze-flutter-project/SKILL.md /path/to/skills/
```

如果使用支持注册 directories 的代理，也可以将本仓库目录添加为 skills 路径。

### 卸载

删除对应的 SKILL 文件：

```bash
rm /path/to/skills/analyze-flutter-project/SKILL.md
```

---

## 🚀 如何使用

安装后，SKILL 会根据你的请求自动触发：

- **自动触发** — 当你说「帮我分析这个 Flutter 项目」时，`analyze-flutter-project` 技能会自动激活
- **手动调用** — 部分代理支持通过 `/skill-name` 命令手动调用
- **跨 SKILL 引用** — 在其他 SKILL 中可以通过名称引用本仓库的 SKILL，实现技能组合

## 🧩 开发新 SKILL

### 目录结构

```
my-new-skill/
└── SKILL.md
```

### SKILL.md 模板

```markdown
---
name: my-new-skill
description: 简短描述技能的用途和触发条件。当用户说「xxx」时触发。
---

# 技能标题

## 铁律 / 约束

## 执行流程

### 1. 第一步

### 2. 第二步

## 输出规范
```

### 设计原则

- **单一职责** — 每个 SKILL 只解决一个问题
- **触发条件明确** — `description` 中写清楚什么场景触发
- **可执行、可验证** — 每一步都是具体操作，结果可检查
- **具体 > 抽象** — 用具体示例代替抽象原则
